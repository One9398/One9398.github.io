View Controller Programming Guide For iOS (一)

发表于 2016-07-25   |   分类于 编程

控制器总览

1.控制器的作用

iOS中ViewControllers分两种:

  • 内容控制器,管理自己根视图上的内容.
  • 容器控制器,管理子控制器和控制子控制器间的切换


1.1视图管理

对于内容控制器来说,控制器管理着属于自己根视图的所有层次视图,控制器通过对其根视图和所有子视图进行强引用来管来每一个视图对象.
而容器控制器只管理自己的根视图以及子控制器的根视图尺寸和位置,不管理子控制器上的视图内容,如系统的容器控制器SplitViewController

1.2数据处理

控制器还管理视图与模型之间的数据处理,将模型数据进行一定加工装载到视图上.但是控制器处理模型数据时要尽量减少对模型数据的管理,应让模型对象自身拥有数据处理的逻辑,然后由控制器进行执行,避免控制器代码的臃肿.


1.3用户交互

作为响应者对象,也具有基本的处理响应链中传递的事件的能力,但多数情况下视图都应自己处理对于的触摸事件或者使用delegate或target-action将事件转交给控制器处理.

1.4资源管理

控制器持有着所有在其根视图上的视图(包括根视图),以及创建的其他对象,所有这些对象的内存都由控制器分配管理.在手机内存紧张时,系统会自动地调用控制器的didReceiveMemoryWarning方法进行临时资源内存的释放,通常可以在此方法中手动释放一些懒加载的对象以及其他不重要但内存占量大的对象.

1.5适应多尺寸

控制器负责视图的应对各个尺寸环境下的呈现方式,比如iPhone的横屏与竖屏两种不同尺寸:水平方向和垂直方向上的Regular Size和Compact Size,控制器应使用Autolayout来合理地布局内容到对应尺寸上,进行布局的动态调整.

2.控制器层次

2.1根控制器

根控制器是控制器层次中的最末端,也是Window对象rootViewController属性指向的对象,来提供初始的视图内容添加到当前的可视窗口上.


2.2容器控制器

容器控制器结合多个子控制器的自定义视图内容组成最终界面来呈现内容,为此需要合理地放置子控制器根视图以及恰当地填充整个容器控制器的根视图界面.

2.3呈现视图控制器

为了在当前控制器下呈现出另一个新的控制器来展示新的内容,常常使用默认的Modal Presentied方式来表现,其中presentedViewContoller表示着呈现出来的新控制器对象,presentingViewController为原来界面的控制器对象.

当有容器控制器存在情况下进行呈现新控制器操作时,UIKit提供合适的上下文来表现:默认选择跟当前控制器层次关系最近的容器控制器来presented新控制器,通常也是窗口的根控制器,当然可允许调整控制器所定义的呈现上下文,进行手动地呈现控制.

3.设计建议

3.1尽可能使用系统支持控制器

iOS系统中存在许多官方定制的控制器如UIImagePickerController,UIPageViewController等,都是为针对解决某一个需求而存在的,在创建自定义控制器前应该在目前系统框架中寻找有否存在能满足需要的系统控制器存在,来优先使用.

3.2保持每个控制器独立

控制器之间不需要过多了解其他控制器的层次关系,而传递数据时通常只需要公开接口调用或者使用委托代理模式进行控制器间的通讯.

3.3尽可能在其根视图布局子视图

将根视图作为容器,合理正确地布局在其之上视图,使用Autolayout布局工具让子视图能随着容器视图的尺寸变化而动态改变布局.

3.4了解数据对象的生命周期

虽然控制器作为视图和模型之间数据传输的中介,但有关模型数据处理和检验最好让模型对象自己实现,统一由控制器调用.

3.5适应性变化

对应不同的设备尺寸和不同尺寸环境,控制器需要使用自建响应变化如系统的尺寸变化和设备方向改变的通知等,支持设备屏幕的size和sizeClass变化.

View Controller Programming Guide For iOS (二)

发表于 2016-07-25   |   分类于 编程

1子类化控制器定制

1.1自定义界面UI

官方推荐使用Storyboard文件以可视化方式进行控制器的UI定制,来对应多种尺寸环境,并且方便快速地调整和查看.


1.2处理用户交互

通常,控制器主要处理交互事件方式有三种:

  • 由视图指定为Target的动作事件和手势事件
  • 系统或者其他对象发送的通知事件处理
  • 作为其他对象的数据源和代理事件处理


1.3运行时展示视图

当控制器加载视图后会调用viewWillAppear:方法让控制器知道将要呈现在屏幕上的视图,然后更新视图布局,展示视图到屏幕上,然后就调用viewDidAppear:.
当动态地添加,删除或修改视图时需要对应地进行约束布局调整.


1.4管理视图布局

当控制器上视图的约束布局变化时,UIKit会通知这些变化的发生,并允许在几处关键的响应过程进行额外的处理工作.

  • 更新控制器和视图的trait collections,
  • 调用控制器的viewWillLayoutSubviews
  • 对当前UIPresentationController 对象调用containerViewWillLayoutSubviews
  • 调用根视图的layoutSubviews,内部遍历子视图进行调用
  • 应用计算后的布局信息到视图
  • 调用控制器的viewDidLayoutSubViews
  • 调用当前UIPresentationController对象的containerViewDidLayoutSubviews方法.

管理布局的建议

  • 使用Autolayout
  • 利用top and bottom layout guides
  • 添加移除视图时对于更新约束
  • 实现动画时临时移除约束


1.5有效地管理内存

控制器中主要进行内存分配和释放的关键方法.

  • init系列的初始化方法: 加载首要的数据
  • viewDidLoad: 装载数据,并配置视图内容
  • didReceiveMemoryWarning: 释放非首要的数据内存
  • dealloc: 释放所有与当前控制器相关的对象内存.


2如何实现容器控制器

2.1设计自定义的容器控制器

容器控制器只应该管理自己视图层次上的视图,再让子控制器管理各自的视图层次.

设计容器控制器的注意点

  • 明确容器控制器和子控制器各自的作用
  • 存在几个同时展示的子控制器
  • 子控制器之间的关系
  • 子控制器如何添加或移除
  • 子控制器尺寸,位置变化的处理
  • 容器控制器对子控制器的UI定制
  • 容器控制器与子控制器之间的通信方式和事件处理
  • 容器控制器的UI定制

以下为系统级容器控制器的例子: NavigationViewController & SplitViewController


2.2Interface Builder配置容器控制器

使用Xcode中UI组件库Container对象,进行子控制器内容视图的占位,加载时同时加载子控制器的视图.


2.3实现自定义容器控制器

实现明确的控制器间子父关系


2.3.1添加子控制器

  1. 使用addChildViewController添加到容器控制器
  2. 添加子控制器的根视图到容器控制器的视图层次中,限定size和postion
  3. 调用didMoveToParentViewController,提供子控制器响应控制器关系改变的机会

注意: didMoveToParentViewController:方法必须在视图层次变化后调用.

2.3.2移除子控制器

  1. 调用子控制器的willMoveToParentViewController方法,传nil
  2. 移除在容器视图上跟子控制器相关的布局约束
  3. 移除子控制器视图
  4. 调用子控制器的removeFromParentViewController完成子父关系的结束

注意: 可以在子类的willMoveToParentViewController方法进行额外自定义操作

2.3.3子控制器间的切换

  1. 准备进行切换的子控制器
  2. 切换时改变子控制视图的层次,设置子控制器切换后的frame
  3. 通知子控制器间切换的完成

2.3.4管理子控制器的界面更新

切换子控制器时,UIKit自动进行按照默认顺序转发外观方法,若需要实现自定义顺序的转发,需要重写容器控制器shouldAutomaticallyForwardAppearanceMethods方法,返回NO;然后手动调用子控制器的beginAppearanceTransition:animated:和endAppearanceTransition方法来通知子控制器.

2.3.5创建容器控制器的建议

  • 让容器控制器只能访问子控制器的根视图
  • 子控制器保持对容器控制器的耦合,使用委托代理将事件转发给容器控制器处理
  • 在容器控制器中首先使用常规的视图,再去嵌入子控制器的根视图.


2.4代理控制子控制器

容器控制器通常会委托一个或者多个子控制器来进行部分外观的配置,配置的方式可以考虑以下两种:

  • 重写childViewControllerForStatusBarStyle或childViewControllerForStatusBarHidden让子控制器决定顶部状态栏风格
  • 使用子控制器的preferredContentSize属性让其确定自己尺寸.


3支持便利性

为了让每一用户即使生理上存在缺陷也能最大程度地使用iOS应用, UIKit的对象默认都accessible,而对控制器仍然在要以下方面保持便利性:

  • 确保每个界面元素accessible,比如标签或者按钮等
  • 确保每个accessible元素提高准确有效的信息


3.1使用VoiceOver功能

开启VoiceOver功能后,界面光标可以移动到特定的元素进行语音告知,主要通过UIAccessibilityScreenChangedNotification通知和UIAccessibilityPostNotification(,)方法,传入特定的控件元素来实现,在布局变化时使用UIAccessibilityLayoutChangedNotification通知来告知用户.


3.2响应VoiceOver手势

UIKit会寻找在响应者链中法实现相应的手势的方法，成为VoiceOver的焦点。如果没有对象实现了对应的方法，UIKit执行系统默认动作。


3.3监听Accessibility通知

使用默认的通知中心注册为可访问性通知的观察者,如UIAccessibilityAnnouncementDidFinishNotification通知和UIAccessibilityVoiceOverStatusChanged告知用户App的状态变化.

4状态的保存和恢复

为了提供App更好的用户体验,iOS控制器提供了App界面状态的存储和恢复功能,即使App在一些特定情况下如内存紧缺,电量不足而中断,能够进行存储配置来记录App退出前界面和数据,保证下次启动App时将中断退出前呈现再给用户.

配置过程

  • 1.对需要具有存储功能的控制器赋值restoration identifiers
  • 2.在应用启动时创建具有restoration identifiers且被应用中断的控制器对象
  • 3.每一个由于存储恢复创建的控制器配置对应数据,保证呈现用户的界面与之前退出的界面一致.


4.1标记需要存储的控制器

设置控制器的restorationIdentifier属性为唯一字符串,默认为nil.并且保证其控制器的所有父控制器也设置了restorationIdentifier,从而使得状态恢复过程中能到从窗口对象开始,一层层查找到需要状态恢复并且呈现在界面上的控制器.
Tip: restorationIdentifier通常设置为控制器类名本身的字符串.


4.2启动时恢复控制器

App启动时,会在AppDelegate中进行具有存储功能控制器需要恢复的定位,并且创建该对象,通过一系列的步骤进行查询和创建.

  • 调用viewControllerWithRestorationIdentifierPath:coder:判断是否存在具有RestorationIdentifier路径的控制器,没有就会返回nil.
  • 然后调用application:viewControllerWithRestorationIdentifierPath:coder:查看没有恢复功能的控制器类
  • 若果存在具有RestorationIdentifier路径的控制器情况下,系统会在启动过程中通过恢复路径加载这些控制器对象.

4.3对控制器状态的编码和解码

针对进行状态保存和恢复的对象,系统都会调用当前对象的encodeRestorableStateWithCoder:和decodeRestorableStateWithCoder:方法,来应用到该对象上,通常所做为保存和恢复所引用的数据对象,其子控制器,当前选择的状态以及配置过的视图.
在编码过程中,需要存储控制器的各个状态数据,而在界面控制器中需要恢复控制器的各个状态数据.

4.4保存和存储控制器的建议

  1. 不要保留所有控制器的状态,仅考虑必要的最先呈现给用户的控制器
  2. 避免在恢复过程中对控制器类进行交换
  3. 使用状态恢复系统推荐的控制器存储恢复方式实现界面的保存和再恢复状态.

Animation

iOS程序中,UIKit和Core Animation为视图提供了丰富的动画来表现视图变化效果.无论是针对UIView还是CALayer对象,在许多属性上提供动画效果.
UIView的可动画属性

  • frame
  • bounds
  • center
  • transform
  • alpha
  • backgroundColor
  • contentStretch

CALayer的可动画属性

  • size
  • center
  • position
  • z-order
  • shadow
  • opacity
  • content

视图的动画代理
通过UIView的setAnimationDelegate:方法就能让动画代理对象根据动画代理方法监听动画开始和结束的情况

- (void)animationWillStart:(NSString *)animationID context:(void *)context;
- (void)animationDidStop:(NSString *)animationID finished:(NSNumber *)finished context:(void *)context;

Tips: 通过设置repeatCount为0.5可以让执行动画视图保存新状态,而不是回到旧的状态.

View Programming Guide For iOS (四)

发表于 2016-07-06   |   分类于 编程

View

视图作为程序中主要的交互对象,自然有许多职责和功能,比如负责布局和子视图的管理, 内容的绘制和动画,以及响应事件的处理.

视图的创建和配置

创建视图
创建视图有两种方式:Nib加载和纯代码加载.其中如果采用Nib加载的方式,又分为加载控制器的Nib文件和加载视图的Nib文件两种.若是加载控制器Nib则加载后控制器会在合适的时机使用和移除其根视图,纯视图的Nib文件加载就需要从NSBundle对象或者UINib对象去手动地创建.


Tag标示视图
对视图的tag属性设置可以用于后续视图的获取,使用viewWithTag:方法,而基于Tag的视图搜索采用深度优先搜索,比遍历视图效率更高.

视图层次的创建和管理

视图添加移除的内存管理
通过addSubview后的视图会自动被父视图强引用,所以在保证视图不被移除父视图的视图层次的情况下单独引用该视图时推荐使用弱引用;通过removeFormSuperView方法从原有视图层中移除视图后,视图会被autorelease一次,如果该视图进入下一次事件循环周期前仍然没有被其他对象强引用,则视图就会被释放掉.


如何隐藏视图
让视图暂时地不可见有两种方式:设置hidden属性为YES;改变aplha属性为0.后一种是具有动画性的属性,可以用于动画,前一种则不会产生动画.

隐藏的视图仍会可能成为事件的第一响应者,因此在隐藏前需要显示地resign第一响应者状态.



如何定位视图层次的视图

  1. 对特定视图使用强引用
  2. 设置视图的tag属性,使用viewWithTag:方法获取



视图的旋转,位移,缩放
针对视图的transform属性进行修改,可以产生视图旋转,位移,缩放的效果,这些效果都是围绕视图的center进行变化.多次对视图transform设置时,不同的变换顺序会产生不同的展示效果.



视图层次中转换坐标

  • 对于UIView对象提供convertPoint:fromView等方法进行不同视图间坐标系的转换.
  • 对于UIWindow对象提供类似convertPoint:fromWindow等方法进行不同窗口间坐标系转换.

如下图当子视图应用了rotation transform后,将子视图基于自己坐标系的矩形转换到父视图坐标系下,生成的区域矩形比原来矩形更大,是在父坐标系下包含子视图原来矩形的最小矩形.

动态调整视图的尺寸/位置

视图布局的变化时机

  • 视图size改变
  • 设备方向的改变
  • 视图相关的sublayers的改变
  • 对视图调用setNeedsLayout或者layoutIfNeed方法
  • 对视图的layer调用setNeedsLayout方法



管理视图布局
iOS程序管理视图布局的方式主要有为设置frame手动管理布局变化的方式和使用autolayout或者Autoresizing规则自动处理布局变化两种.

  1. 当视图尺寸变化时,一定会自己调用layoutSubviews进行子视图frame的自动调整,通过重写该方法可以覆盖系统对子视图frame的调整,对其进行自定义设置,比如:

    • 调整自己任何子视图的尺寸和位置
    • 添加移除子视图或者Core Animation图层
    • 调用子视图的setNeedsDisplay或setNeedsDisplayInRect:方法迫使子视图重新绘制

  2. 利用autolayout布局引擎和Autoresizing规则能够让视图自动处理自己尺寸和位置的变化来应对父视图新的size.不过最终的布局效果仍可以在layoutSubviews中进行调整.



视图与层
每一个View都有一个layer提供内容,可以在View的+layerClass方法指定特定的layer类,但一旦指定,该layer的实例对象也成为View的delegate,不应该再去改变它们的关系.

视图的自定义

如何实现自定义
由于系统控件的视图不能总是满足产品需求,开发者往往需要去实现自定义的视图,对视图进行自定义后,就完全控制了视图内容的呈现和事件处理,也因此需要特别注意几个地方,让自定义视图的表现能像系统控件一样稳定,功能能满足自己的需求.

  • 提供合适的初始化方法.如果视图只能被纯代码地创建,则需要重写initWithFrame方法;而要用nib加载视图,则需要重写initWithCoder:方法进行视图初始化.
  • 实现dealloc方法提供视图数据的清理
  • 若要自定义绘制内容,则重写drawRect:方法
  • 若存在子视图,则需要要初始化时进行添加,并且在layoutSubviews方法中布局.
  • 提供基于Touch的事件处理



自定义视图绘制
视图首次绘制时会调用drawRect:方法,方法里UIKit会自动创建一个限定区域的图形上下文,可通过UIGraphicsGetCurrentContext方法获得.



自定义视图事件响应
视图对象由于继承UIResponder类,也因此具有了处理touch事件的能力.
UIApplication提供两个方法beginIgnoringInteractionEvents和endIgnoringInteractionEvents让视图在临时一段时间内不接受touch事件.
在事件传递过程中UIKit会调用hitTest:withEvent:和pointInside:withEvent:方法来决定事件是否在给定视图bounds内容,你可以重写这些方法来阻止子视图接收touch时间.



自定义视图的清理
当自定义视图被释放时,任何视图内容相关资源和数据应该在dealloc方法中释放掉,而不是执行其他任务性的操作.

习惯的规则

View Programming Guide For iOS (二)

发表于 2016-07-04   |   分类于 编程

View and Window Architecture

UIVIew和UIWindow类,在管理布局和内容呈现上提供了复杂的实现,理解这些实现是怎么回事,更有利于明白自定义视图在程序中的表现以及如何正确地响应程序de变化.

View Architecture Fundamentals

UIKit视图的内容渲染和动画其实是由Core Animation Layers 处理,每一个UIKit的视图对象都依靠对应一个layer对象来管理视图内容的后备存储器(backing store)和动画处理.当你需要对动画或者渲染需要进一步控制和自定义时,就需要在视图的layer上进行操作.

涉及视图绘制时,Core Animation layer对象在执行绘制代码时会产生一定性能消耗,应该尽可能地少使用绘制相关的代码,而绘制后的结果会被Core Animation缓存起来,尽可能地重用.这样的重用机制主要是避免绘制周期时更新视图内容时重复绘制生成的昂贵性能代价.并且绘制内容的重用在执行动画中也十分重要,因为重用比创建内容的耗费的性能要小得多.

View Hierachies and SubView Management

View除了提供自身内容,也能作为其他视图对象的容器,提供其他视图的内容,成为自己的子视图,形成子父级的视图层次关系.

向View的subViews数组添加子视图时,subViews中越是最后的视图对象,它的内容就越靠前地呈现在父视图的上方,从而可能遮挡住其他子视图的内容.而子父视图的关系让父视图能在改变自身的尺寸,位置或者透明度等属性时影响到它的所有子视图.

层次级的视图管理也决定了程序中事件的响应处理方式: 当在一个特定视图上发送Touch事件,系统会将事件的信息包装在UIEvent对象中,发送给被touch的视图对象,让其完成对事件的处理,如果该视图不处理该Touch事件,事件就会被传递到它的父视图进行查询是否处理事件,如果它的父视图也不处理该事件,事件就会一层层地向位于更高视图层次的视图传递,这样的过程形成的就是一个响应链 (responder chain),并且每一个视图都属于响应者对象,其中控制器也是响应者对象,这也表示这视图可以直接将事件传递给控制器,让其处理事件.如果没有任何对象处理一个事件,那它将会传递到响应链的最顶层Application对象上,如果Application对象也没处理该事件,则事件就会因没有对象来处理而丢弃.

The View Drawing Cycle

UIView采用按需绘制模型来展现内容: 当视图将首次出现在屏幕上时,系统执行绘制代码来绘制内容并且渲染到屏幕上,然后对绘制的结果内容生成快照缓存,以便后期的视图内容重用.这样以后只要视图的内容不被改变,绘制代码就不会再被执行,而是使用首次绘制后留下的缓存结果,但是一旦视图的内容发生改变,系统就会收到绘制需要执行的通知,执行新的绘制代码过程,并且生成新绘制后的快照缓存.

视图内容的变化后,调用setNeedsDisplay相关方法,会通知系统关于视图的绘制和内容变化将延时到下一个运行循环的绘制阶段中进行,这样就可同时改变不同视图的内容甚至结构层次,然后在屏幕上进行这些变化的同步.

Note: 改变视图的几何特征是不会导致内容的重绘,而视图的几何特征由view的contentMode属性来表现,设置contentMode后的行为表现就是操作内容绘制后的快照结果, 让其在视图限定的区域内进行伸缩和移动.这样的行为并不会创建的绘制内容,是比较高效的.

Content Modes

针对视图当尺寸和位置变化时需要重新呈现视图内容所对应的一种对视图内容的回收使用的优化.

每个视图都有一个内容模式，控制如何回收使用它的内容来响应视图中的几何特征的变化以及是否回收利用其全部的内容。当第一个视图显示时，它呈现它的内容像往常一样，结果被捕获在一个基本的位图中。之后，在视图几何外形的改变并不一定会导致重新创建位图,在方式属性的值决定旧的位图是否应该缩放或者移动来适应新的边界.

content mode被应用的时机

  1. 视图的frame或者bound中的width/height发生变化
  2. 对视图的transform属性进行新的trasnform设置

content mode 主要模式

enum UIViewContentMode : Int {
    case ScaleToFill
    case ScaleAspectFit
    case ScaleAspectFill
    case Redraw             // important
    case Center
    case Top
    case Bottom
    case Left
    case Right
    case TopLeft
    case TopRight
    case BottomLeft
    case BottomRight
}

值得注意的一点是,其中UIViewContentMode中Redraw表示视图在发生缩放和调整大小时不采用系统默认使用旧的绘制快照来调整视图内容变化,而是重新执行绘制代码.这样场景适用于当自定义视图需要自己控制内容变化时绘制方式,通过让系统强制调用drawRect相关方法来响应几何特征的变化.

View Geometry and Coordinate Systems

Coordinate Systems Orientation

iOS中,UIKit中的坐标系以设备右上角为原点,向右为x轴正方向,向下为y轴正方向;而Core Graphic 和OpenGL ES所采用的坐标系是以左下角为原点,向上为y轴正方向,向右为x轴正方向.

The Relationship of the Frame, Bounds Center

frame和center都是根据视图以它父视图坐标系进行size和location的设定.而bounds表示视图根据自己的坐标系指定size和origin.
iOS程序中主要使用frame和center来操作视图的几何.而当视图的transform属性不为Identity Transform时,访问frame值是不准确,存在偏差的;应该使用center来获得图像有效中心位置.

frame,bounds,center的变化关系
当frame改变时,bounds的size会对应变化,center也会变化;当center变化时frame的origin会变化;当bounds的size改变时,frame的size会对应变化.

若一个子视图的frame超过了父视图本身的内容区域,超出的那部分默认情况下仍会渲染绘制出来,但是在那部分内容无法进行事件的响应接收和处理.另外,也可以设置父视图的clipToBounds为Yes,进行超出区域内容的裁剪.

Coordinate System Transformations

利用view的transform属性可以快速改变视图的尺寸,位置,角度等表现效果,而设置的所有的transform都是围绕视图的center进行视图几何特征的变化.

由于每个子视图的坐标系统都基于他们的祖先视图的坐标系统建立,若改变一个父视图的transform,则会影响到所有其子视图的布局.

一旦视图的transform被设置,其的frame的值会变得无效,需要使用bounds和center来得到有效的尺寸和位置数据,但其子视图的frame是根据父视图的bound而生成的,所以子视图的frame仍是有效的.

The Runtime Interaction Model for Views

当前用户在屏幕上某个视图进行touch时,一系列事件的传递就从硬件到软件发送.

  1. 用户触屏屏幕某个视图
  2. 硬件向UIKit框架传达touch事件
  3. UIKit框架将事件打包成UIEvent对象,分发给最合适处理该事件的视图,一般是touch所在位置的视图.
  4. 视图执行处理touch事件的代码
  5. 视图的几何特征如果变化,UIKit就会更新它的子视图.
  6. UIKit对被标记了重绘标记的视图进行绘制.
  7. 更新后的视图与屏幕其他的视图发送给图像硬件去显示.
  8. 图像引擎将渲染的内容传输到屏幕上

在整个处理过程中,若使用自定义视图,主要有以下几个关键处理点

  • 添加事件处理方法,touchesBegan:withEvent:系列方法或者使用手势事件
  • 实现layoutSubViews
  • 实现drawRect

Tips for Using Views Effectively

View章节最后,官方给了几个高效使用视图的建议.

  • 尽量减少自定义的绘制,自定义绘制难免需要一些性能消耗,尽可能使用现有的系统视图控件来组合成所需要的视图层次.
  • 利用Cotent Modes,减少已渲染的视图的重绘,并且尽量避免设置为UIViewContentModeRedraw;在需要视图再重新绘制时使用setNeedsDisplay相关方法来更新视图的内容.
  • 尽量让视图保持不透明,设置视图的opaque属性为YES,这样UIKit就不需要再渲染该视图下面所遮挡的其他视图的内容.
  • 在滚动行为中调整视图的绘制行为,尽量减少在滚动时内容的绘制操作来保证滚动的流畅性,在滚动停止时进行视图内容的更新.
  • 不要通过在系统标准视图上嵌入子视图来定制自定义视图,极有可能影响到原有控件的绘制和功能.相对的,可以在UIView上通过组合系统视图来完成自定义视图.
