《怪诞行为学》

文摘

关于拖延

面对长远目标的拖沓,给人以自设底线的机会,选择喜欢的行动方式,来完成目标.



关于攀比,比较

我们用相对方法看待我们的决定,就近与现成的其他选择作比较.

人心不足蛇吞象.唯一的解决方法就是打破相对论的怪圈.



关于生活

我们都是棋盘上的小卒子,对棋局里的多种力量认识不多.我们通常认为自己稳坐在驾驶座上,对自己的决策,对自己生活的方向具有至高无上的操控能力;但是,这只是我们的一厢情愿,它反映的是我们的欲望—我们对自身想当然的认识,而不是事实.



关于消费

我们做出的种种决定都是基于自己的基本价值判断-喜欢或不喜欢.

从自己的个人生活来说,我们有能力积极改进自己的非理性行为,从认识自己的脆弱开始.

特别关注所作出的首次决定,它的长期效应会渗透到未来我们所作的许多决定里,必须给予足够的重视.



关于诚实

当自己处于某种条件下,个人利益与道德标准发生对立,我们容易向现实屈服,以一己之私看待事件,从而变得不诚实—我们要认识到这种可能,认识到自己的弱点,尽力避免这种情况的发生.

在做事的诚信上,我们更轻易地在非金钱事物上作弊,为了尽量自控需要明确认识到非货币财货与自己作弊倾向的关系.



关于社会与市场

我们同时生活在两个不同的世界里-其中一个世界由社会规范主导,另一个则由市场规范来定制法则.

一旦社会规范与市场规范发生碰撞,社会规范就会退出,由市场社会主导.

现实中多一些社会规范,少一些市场规范,生活就会变得更惬意,更有创造力,更充实,而且更加有趣.



关于情绪

我们需要探索情绪自身的两个方法,需要了解冷静状态,也需要了解激情状态,明白之间的不同.



关于拥有

我们总是把注意力集中到自己会失去什么,而不是会得到什么.

用”非拥有的心态”来看待每一项交易,把自己和感兴趣的物件拉开适当的距离.

对世间万物,尽量待之以平常心.



关于选择

在现代民主制度下,困扰人们的不是缺乏机会,而是机会太多,令人眼花缭乱.

Window

Window对象除了作为装载视图的容器这个作用外,还把负责touch event的分发给最适合处理事件的视图或者其他对象.

所使用Window对象能做的事

  • 将基于窗口坐标系的点和矩形与其他坐标系之间进行值的转换
  • 根据窗口变化发出的系统通知进行特定操作的执行

创建和配置Window的要点
尽可能地懒加载Window,在需要呈现的时候创建,并且保持对Window对象的强引用.

向Window上添加内容
通过addSubView添加在Window上的视图被称为Window的root View,而设置window的rootViewController属性后,rootViewController的视图自动被添加到Window上,称为root View.

改变Window等级
每个Window对象都有一个windowLevel,决定着多个Window之间显示的优先等级,level越高则window就显示得越前面,而覆盖住level低的Window上的内容.以下为系统提供的windowLevel的取值.

UIWindowLevelNormal // 默认值
UIWindowLevelAlert  
UIWindowLevelStatusBar

Window变化的监听
当Window隐藏或者显示等状态发生变化时,系统会发送Window相关的通知,给iOS程序提供机会进行对于的操作.

UIWindowDidBecomeVisibleNotification
UIWindowDidBecomeHiddenNotification
UIWindowDidBecomeKeyNotification
UIWindowDidResignKeyNotification

UIWindowDidBecomeKeyNotification和UIWindowDidResignKeyNotification的通知监听着程序的keyWindow的变化,keyWindow表示当前能接收键盘事件和其他非touch事件的窗口,而keyWindow接收到touch事件不会有与touch位置相关的值.

keyWindow永远只能出现一个,但可以相互切换.

View Programming Guide For iOS (2)

发表于 2016-07-04   |   分类于 编程

View and Window Architecture

UIVIew和UIWindow类,在管理布局和内容呈现上提供了复杂的实现,理解这些实现是怎么回事,更有利于明白自定义视图在程序中的表现以及如何正确地响应程序变化.

View Architecture Fundamentals

UIKit视图的内容渲染和动画其实是由Core Animation Layers 处理,每一个UIKit的视图对象都依靠对应一个layer对象来管理视图的后备存储器(backing store)和动画处理.当你需要对动画或者渲染需要进一步控制和自定义时,就需要在视图的layer上进行操作.

涉及视图绘制时,Core Animation layer对象hi执行绘制代码时会产生一定性能消耗,应该尽可能地少使用绘制相关的代码,而绘制后的结果会被Core Animation缓存起来,尽可能地重用.这样的重用机制主要是避免绘制周期时更新视图内容时重复绘制生成的昂贵性能代价.并且绘制内容的重用在执行动画中也十分重要,因为重用比创建内容的耗费的性能要小得多.

View Hierachies and SubView Management

View除了提供自身内容,也能作为其他视图对象的容器,提供其他视图的内容,成为自己的子视图,形成子父级的视图层次关系.

向View的subViews数组添加子视图时,subViews中越是最后的视图对象,它的内容就越靠前地呈现在父视图的上方,从而可能遮挡住其他子视图的内容.而子父视图的关系让父视图能在改变自身的尺寸,位置或者透明度等属性时影响到它的所有子视图.

层次级的视图管理也决定了程序中事件的响应处理方式: 当在一个特定视图上发送Touch事件,系统会将事件的信息包装在Event对象中,发送给被touch的视图对象,让其完成对事件的处理,如果该视图不处理该Touch事件,事件就会被传递到它的父视图,交个它的父视图处理,如果它的父视图也处理该事件,事件会一层层地向上传递,以此类推进行对事件的接收判断和处理,这样的过程形成的就是一个响应链*(responder chain)*,并且每一个视图都属于响应者对象,其中控制器也是响应者对象,这也表示这视图可以直接将事件传递给控制器,让其处理事件.如果没有任何对象处理一个事件,那它将会传递到响应链的最顶层Application对象上,然后丢弃掉.

The View Drawing Cycle

UIView采用按需绘制模型来展现内容: 当视图首次出现在屏幕上时,系统执行绘制代码来呈现内容,然后对绘制的结果内容生成快照,以便后期的视图内容重用.这样以后只要视图的内容不被改变,绘制代码就不会再被执行,而是使用首次绘制后生成的快照结果,但是一旦视图的内容发送改变,系统就会收到绘制需要变化的通知,执行新的绘制代码过程,并且保存新绘制后的快照结果.

视图内容的变化后,调用setNeedsDisplay相关方法,会通知系统关于视图的绘制和内容变化将延时到下一个运行循环的绘制阶段中进行,这样就可同时改变不同视图的内容,结构层次,然后在屏幕上进行这些变化的同步.

Note: 改变视图的几何属性,不会导致内容的重绘,而视图的几何属性由view的contentMode属性来表现,设置contentMode后的行为表现就是操作内容绘制后的快照结果, 让其在视图限定的区域内进行伸缩和移动.这样的行为并不会创建的绘制内容.

Content Modes

针对视图当尺寸和位置变化时需要重新呈现视图内容所对应的一种对视图内容的回收使用的优化.

每个视图都有一个内容模式，控制如何回收使用它的内容来响应视图中的几何变化和是否使用其全部的内容。当第一个视图显示时，它呈现它的内容像往常一样，结果被捕获在一个基本的位图中。之后，在视图几何外形的改变并不一定会导致重新创建位图,在方式属性的值决定旧的位图是否应该缩放或者移动来适应新的边界.

content mode被应用的时机

  1. 视图的frame或者bound中的width/height发生变化
  2. 对视图的transform属性进行新的trasnform设置

content mode 主要模式

enum UIViewContentMode : Int {
    case ScaleToFill
    case ScaleAspectFit
    case ScaleAspectFill
    case Redraw             // important
    case Center
    case Top
    case Bottom
    case Left
    case Right
    case TopLeft
    case TopRight
    case BottomLeft
    case BottomRight
}

值得注意的一点是,其中UIViewContentMode中Redraw表示视图在发生缩放和调整大小后不会使用到原有的绘制快照,而是重新执行绘制代码.这样场景适用于当自定义视图需要自己控制内容变化时绘制方式,通过让系统强制调用drawRect相关方法来响应几何图形变化.

View Geometry and Coordinate Systems

Coordinate Systems Orientation

iOS中,UIKit中的坐标系以设备右上角为原点,向右为x轴正方向,向下为y轴正方向;而Core Graphic 和OpenGL ES所采用的坐标系是以左下角为原点,向上为y轴正方向,向右为x轴正方向.

The Relationship of the Frame, Bounds Center

frame和center都是根据视图以它父视图坐标系进行size和location的设置.而bounds表示视图根据自己的坐标系指定size和origin.
主要使用frame和center来操作视图的几何.而在视图的transform属性不为Identity Transform时,使用的frame值是不准确的,应该使用center来获得图像有效中心位置.

frame,bounds,center的变化关系
当frame改变时,bounds的size会对应变化,center也会变化;当center变化时frame的origin会变化;当bounds的size改变时,frame的size会对应变化.

若视图的frame若是超过了父视图的frame,超出内容的那部分默认情况下仍会渲染绘制出来,但是在那部分内容无法进行事件的响应处理;也可以设置父视图的clipToBounds为Yes,进行超出内容的裁剪.

Coordinate System Transformations

利用view的transform属性可以快速改变视图的尺寸,位置,角度等表现效果,应用后所有的transform都是围绕视图的center而变化.

每个子视图的坐标系统都基于他们的祖先视图的坐标系统建立,若改变一个视图的transform,就会影响到其子视图.

一旦视图的transform被设置,其的frame的值会变得无效,需要使用bounds和center来得到有效的尺寸和位置数据,但其子视图的frame仍是有效的.

The Runtime Interaction Model for Views

当前用户在屏幕上某个视图进行touch时,一系列事件的传递就从硬件到软件发送.

  1. 用户触屏屏幕某个视图
  2. 硬件向UIKit框架传达touch事件
  3. UIKit框架将事件打包成UIEvent对象,分发给最合适处理该事件的视图,一般是touch所在位置的视图.
  4. 视图执行处理touch事件的代码
  5. 视图的几何特征如果变化,UIKit就会更新它的子视图.
  6. UIKit对被标记了重绘标记的视图进行绘制.
  7. 更新后的视图与屏幕其他的视图发送给图像硬件去显示.
  8. 图像引荐将渲染的内容传输到屏幕上

在整个处理过程中,若使用自定义视图,主要有以下几个关键处理点

  • 添加事件处理方法,touchesBegan:withEvent:系列方法或者使用手势事件
  • 实现layoutSubViews
  • 实现drawRect

Tips for Using Views Effectively

View章节最后,官方给了几个高效使用视图的建议.

  • 尽量减少自定义的绘制,自定义绘制难免需要一些性能消耗,如果不是万不得已使用现有的视图来组合成所需要的视图层次.
  • 利用好Cotent Modes,减少已渲染的视图的重绘,尽量避免设置为UIViewContentModeRedraw,在需要视图再重新绘制时使用setNeedsDisplay相关方法来更新视图的内容.
  • 尽量让视图保持不透明,设置视图的opaque属性为YES,这样UIKit就不需要再渲染该视图后面的内容.
  • 在滚动中调整视图绘制行为,尽量减少在滚动时内容的提供来保证滚动的流畅性,在滚动停止时进行视图内容的更新.
  • 不要通过在系统标准视图上嵌入子视图来定制自定义视图,可以在UIView上通过组合系统视图来完成自定义.
